# 稳定币支付聚合器 / Stablecoin Payment Aggregator

> 聚合多种稳定币支付，商户只需接受一种，自动路由兑换

## 解决什么问题

Sui 生态中流通着多种稳定币（USDC、USDT、AUSD、本地稳定币等），消费者钱包中持有的稳定币种类各不相同，商户无法也不应逐一集成支持。目前商户要么只接受单一稳定币流失客户，要么承担多币种管理和兑换的额外成本。稳定币支付聚合器充当统一支付层，消费者用任意稳定币支付，系统通过 DEX 自动兑换为商户指定的单一稳定币，商户只需管理一种币种，同时不限制消费者的支付选择。

## 核心功能

- 多稳定币聚合：支持主流稳定币（USDC、USDT、AUSD 等）统一支付
- 最优路径兑换：自动比较 DEX 池的兑换率和滑点，选择最优兑换路径
- 商户一键结算：商户设定目标稳定币，所有收入自动兑换并汇总结算
- 滑点保护机制：设置最低输出金额保护，防止因流动性不足导致的过大滑点
- 费率透明化：每笔交易展示兑换率、滑点和协议手续费明细，消费者确认后执行

## 使用的 Sui 原语

| 原语 | 用途 |
|------|------|
| PTB | 原子化执行"消费者支付稳定币 → 路由比较 → 最优兑换 → 商户到账" |
| Coin 标准 | 统一处理多种稳定币类型的输入和输出 |
| 共享对象 | AggregatorRouter 作为共享对象处理并发支付 |
| 动态字段 | 存储支持的稳定币列表、路由配置和费率 |
| 对象所有权 | MerchantConfig 对象绑定商户的结算偏好 |

## 实现方案

### 智能合约 (Move)
- `aggregator_registry` 模块：注册支持的稳定币类型和 DEX 路由
- `payment_processor` 模块：处理消费者支付，触发兑换和结算
- `route_optimizer` 模块：比较多条兑换路径的输出金额，选择最优路径
- `merchant_settlement` 模块：管理商户结算地址和目标稳定币配置
- 关键数据结构：`PaymentRequest { payer: address, input_coin: Coin<T>, output_coin_type: Type, merchant: address, min_output: u64, deadline: u64, route_hint: vector<RouteHop> }`

### 前端 / 后端
- 技术栈：Next.js + @mysten/dapp-kit + Sui TypeScript SDK
- 关键页面：Consumer Payment（消费者支付页）、Merchant Dashboard（商户面板）、Route Analytics（路由分析）、Supported Coins（支持的稳定币管理）、Transaction History（交易历史）
- 后端提供路由预计算、汇率缓存和交易监控
- 支持 REST API 和 JavaScript SDK 两种接入方式，方便第三方集成

## 技术架构

商户在前端注册并设置目标稳定币（如 USDC）和结算地址。消费者在支付时，前端调用后端路由预计算接口，获取当前最优兑换路径（比较 DeepBook、Cetus 等多个 DEX 池的兑换率）。确认支付后，系统构建 PTB：从消费者地址扣除输入稳定币 → 按最优路径依次兑换 → 确保输出金额不低于最低限额 → 扣除协议手续费 → 将净额转入商户地址。整个兑换和结算在一个 PTB 中原子化完成，消费者无感知，商户只收到目标稳定币。后端持续监控各 DEX 池流动性，动态更新路由策略。
